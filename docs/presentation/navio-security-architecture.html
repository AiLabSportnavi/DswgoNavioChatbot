System Overview

A controlled, isolated request flow

Every interaction follows a single, validated path. The browser never holds secrets or talks to the AI directly — the Navio backend is the sole, hardened gateway between users and Azure OpenAI.

🔒 Secure communication

End-to-end TLS on every hop. Credentials and the system prompt live only on the server — never shipped to the browser.

➡️ Controlled request flow

One sanctioned path: validate → authorize → call Azure → return. No client ever reaches the model directly.

🧩 Component isolation

Frontend, backend, and AI provider are cleanly separated, so a fault or attack in one tier cannot compromise the others.

Security Architecture

Origin restriction — the first gate

Before any logic runs, the backend verifies where the request came from. Only the official SportNavi domains are permitted to embed and call Navio.

# Backend allow-list (enforced server-side)
ALLOWED_ORIGINS = "https://sportnavi.de,
                 https://www.sportnavi.de"
  • ✓Browser origin validation — CORS policy rejects requests from any domain not on the list.
  • ✕Prevents unauthorized embedding — third-party sites cannot clone or hijack the widget.
  • ✓Reduced abuse surface — scrapers and bots calling from foreign origins are stopped at the door.
  • ✕Domain-level protection — the AI is reachable only through SportNavi's own properties.

Input Protection

Bounded input — a security control

A hard cap on message size keeps payloads predictable. Input limits are not a UX nicety — they are a deliberate defense against resource exhaustion and abuse.

2,000
Maximum characters per message

Validated at the API boundary — oversized requests are rejected before any processing.

MAX_MESSAGE_CHARS = 2000  # rejected if exceeded

🛡️ No oversized payloads

Megabyte-scale junk requests can't reach the model or the database — they're refused at validation.

📊 Predictable resource use

Bounded inputs mean bounded memory, tokens, and latency per request — capacity planning stays accurate.

🎯 Reduced attack surface

Caps blunt payload-stuffing and prompt-bloat attempts aimed at exhausting AI token budgets.

⚙️ Improved reliability

Consistent request sizes keep the service stable under load and simplify failure handling.

Conversation Memory

A bounded, short-lived memory window

Navio remembers only the last 10 turns. Context is intentionally limited — a privacy-conscious choice that also controls cost and performance.

🔐 Privacy-conscious

Less conversation data is held in context, narrowing what's ever processed at once.

📐 Controlled context

A fixed window keeps prompt size and behavior consistent over long chats.

💶 Cost management

Bounded history caps the tokens billed on every single request.

⚡ Performance

Shorter context means faster responses and lower memory pressure.

MAX_HISTORY_TURNS = 10  # oldest turns drop off → reduced data retention

Bot Protection

Cloudflare Turnstile — proving a human

Before a session is issued, the visitor is verified as a real browser. Turnstile mitigates automated attacks without friction-heavy CAPTCHAs.

BEFORE · no bot gate

Scripts and headless browsers hit endpoints freely — spam, scraping, and cost-draining traffic get through.

AFTER · Turnstile enabled

Only verified humans receive a session token. Automated traffic is filtered before it ever reaches the AI.

✓ Human verification
✓ Bot mitigation
✓ Automated-attack prevention
✓ Abuse protection

Session Security

Signed, expiring visitor sessions

Verified visitors receive a cryptographically signed session token with a short lifetime. Integrity is server-validated on every request — and the secret never leaves the server.

# Secrets stay server-side — never in the browser
SESSION_SECRET = <SECRET> # [REDACTED]
SESSION_TTL_MIN = 30
  • ✓Session integrity — HMAC signing makes tokens tamper-evident.
  • ✓Visitor trust validation — only Turnstile-verified visitors get a token.
  • ⏱Session expiration — a 30-minute TTL limits the window of validity.
  • ✓Reduced hijacking risk — short-lived, signed tokens shrink the attack window.
  • ↻Secure visitor lifecycle — expired sessions re-verify cleanly.

Distributed Architecture

Shared quotas across every instance

When scaled horizontally, all application instances enforce rate limits against a single shared Redis store — so quotas hold no matter which worker handles a request.

⚡ Shared rate limiting

One source of truth for counters — no per-instance blind spots.

🔁 Multi-worker consistency

Quotas hold across all workers and instances simultaneously.

📈 Horizontal scalability

Add instances freely; limits remain correct and global.

✅ Reliable enforcement

Atomic counters make quota decisions deterministic under load.

Note: Navio runs in-memory limiting by default for single-instance simplicity and switches to Redis automatically when a distributed deployment is configured.

Data Residency

European hosting, European data

Navio's infrastructure is deployed in EU-hosted Azure regions. For organizations operating within Europe, regional processing is a foundational trust and governance requirement.

  • EUEuropean hosting strategy — compute and AI inference run within EU-hosted Azure infrastructure.
  • 📍Data residency — processing stays in-region, aligning with European governance expectations.
  • 🌍Regional processing — requests are handled close to the audience they serve.
  • 🏛️Enterprise governance — regional control supports internal policy and procurement requirements.
  • 🤝Customer trust — a clear, European-first posture for European organizations and their users.

Why it matters: Organizations operating within Europe increasingly require that personal and operational data be processed within the region. EU-hosted infrastructure makes that posture explicit and auditable.
